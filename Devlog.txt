9/4/26
FIXED IT.

It was because my bottom to top loop was indexing wrong. Turns out this moron forgot how unsigned integers work and was letting his iterator variable decrement all the way down to -1 like so:
    for(size_t i = old_grid->height-1; i > -1; i--) {
meaning it wrapped back around to MAX_INT, causing all sorts of issues. Basically just make it stop at 1, and then subtract one when using it in a method call, like so:
    for(size_t i = old_grid->height; i > 0; i--) {

10/4/26
Created pixel_sim.h and pixel_sim.c.
Decided that the pixel data would be stored as a 'typedef uint32_t pixel_data'.
The data would be stored on a bit level in the following format:
+------------+---------+-----------+-----------+--------+
|    Type    | Variant | VelocityX | VelocityY | Health |
|000000000000|  0000   |   00000   |   00000   | 000000 |
+------------+---------+-----------+-----------+--------+
Type is the type of the pixel (sand, water, wood, stone, acid, etc).
Variant  is used for colouring. For example, even the same chunk of wood can have pixels of different shades ofbrown to break up the monotony. Moreover, there should also be a separate variant for burnt wood.
VelocityX and VelocityY are self explanatory.
Health represents the health (or lifetime) of the pixel at the current moment.
Ideally we would want to have at least one of these values (probably type) always guaranteed to be non-zero, so that we could use 0 as a sentinel value to represent an empty pixel

The idea with this system is that we can store data that relates to an entire pixel type (flammability, weakness to acid, etc.) in a lookup table and does not have to be stored by each individual pixel. Then, we could make a giant switch-based function (like we already do for component types etc) to determine what happens to each pixel type each frame. Then, when a specific branch of the function, we use the data in the pixel_data alongside the values in the lookup table to update the pixel.

Speaking of updates, I finally found that long nolla games dev talk video on youtube: https://www.youtube.com/watch?v=Mr3E_voEG8Y. I changed my update calls so it was more in line with what they said, instead of alternating between going bottom to top and top to bottom, I alwawys now go bottom to top, and instead I now alternate the direction of rows. E.g. on the first row of a pass we go left to right, then right to left, and then left to right and so on, and then on the next iteration we go right to left, left to right, right to left and so on.This did have a slight effect, in the original version it would always go left then right, but (I think????) in this version it is fairly even.

I changed entity IDs like I had been meaning to do for a while, no longer wasting 2^16 possibilities. Entities are now uint32_t as opposed to int32_t, and now just 0 is the reserved value to mean an uninitialised entity (should probably add comments in the code explaining this). This wastes far fewer values and possibly opens them to being compressed to a smaller uint16_t if I feel we don't need more than 2^16 entities in the final project.

I also split the grid into colours, parents, and data maps, instead of just colours and parents. The purposes of these are:
colours: stores the colours of the pixels (could be removed and just write directly to the pixel renderer in future?)
parents: stores the parent rigidbody of a pixel if it has one. Thanks to the changes to entities, 0 is used to represent a parentless pixel
data: stores the pixel_data of each pixel. 0 is also used a sentinel value for an empty pixel (see above for reasoning)

Currently only colours and parents are actually used properly. data is not used since we currently only have a primitive system that only supports a 'sand' pixel type, and I haven't had time to implement something better. In the pixel_sim.h/c files, I have added some bitwise getters and setters to get the values within a pixel_data uint32_t, so those will be used in the future. I also tried implementing a lookup table with enum indecies, but gcc complained about it idk why, so I commented it out (the enum indecies are still there though).

11/10/26:
Added a very basic input manager API. This is so I don't have to keep including GLFW everywhere so that if I every decide to switch away from it there's only one point of failure instead of having to search for random references across my program. Currently only supports A-Z keys and the three mouse buttons (because I'm lazy). Since GLFW only keeps PRESS and RELEASE state for each of its keys, my internal representation is a bit more detailed:
    KEY_JUST_PRESSED: state switched to PRESS this frame
    KEY_PRESSED: state was PRESS the previous frame and this frame
    KEY_JUST_RELEASED: state switched to RELEASE this frame
    KEY_RELEASED: state was RELEASE the previous frame and this frame
All keys are initialised to be KEY_RELEASED and their state is updated through callbacks to GLFW.
The input handler also holds the current cursor position, also updated through a callback

Actually added working buttons to the game. They use smaller versions of the cargo texture that used to be the background (laziness again). I can't seem to figure out how to change their colour (for future). The button component struct was updated so it also contains the arguments passed into the function as a void *.
Updated rendering so that all textures are now rendered scaled to the pixel size. Idk about this one, may change in the future.

Implemented a sort-of event system for updating the pixel grid. The way it works is whenever something fires to update the pixel grid (adding/removing pixels) in the current frame, a struct made up of a callback function that executes that update and its arguments is added to a queue. The callback function has the following signature:
    typedef void (*pixel_func)(pixel_func_args* args);
And the arguments is a struct that looks like this:
    typedef struct {
        ivector2 cursor_pos;
        grid_buffer *gbuf;
        plaza *p;
    } pixel_func_args;
So the added struct looks like this:
    typedef struct {
        pixel_func func;
        pixel_func_args *args;
    } pixel_op_callback;
Then at the start of the pixel_system_update function, the queue is popped and all of the callback functions' operations are carried out on the current pixel grid, so that it is in its newest state when updates occur in the pixel system. There are currently only two defined callback functions:
    void erase_pixels_callback(pixel_func_args *args);
    void add_sand_callback(pixel_func_args *args);
Pretty self explanatory stuff. The second could be extended to just 'add_pixel_callback' since I don't think functionality will change between adding different pixels. We will just need to modify pixel_func_args to also include the pixel_data as an argument.
